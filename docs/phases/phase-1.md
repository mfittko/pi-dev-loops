# phase-1 durable plan

## Status

Completed

## Objective

Normalize the imported assets without changing their intended workflow behavior, and remove the first obvious blockers to reuse.

## Why this phase exists now

Phase 0 established the docs-first workflow convention. Phase 1 now uses that foundation to classify the imported assets honestly and remove the source-repo assumptions that currently block safe reuse claims.

## In scope

- refine this phase doc into the durable source of truth for Phase 1
- classify each imported asset as one of:
  - ready to globalize now
  - reusable after moderate refactor
  - keep local until split into base + overlay
- add one narrow regression test surface for imported blocker phrases
- normalize only the current known blockers:
  - `repo-wiki` wording in `skills/copilot-dev-loop/SKILL.md`
  - required references to companion skills that do not exist in this repo
  - hardcoded reviewer identity in `agents/review.agent.md`
  - stale `docs/plans/` references in imported agent prompts
- keep coordinator cleanup narrow: remove stale assumptions and classify the remaining workflow-policy coupling honestly

## Explicit non-goals

- no base/overlay architecture split yet
- no extra durable inventory doc separate from this phase doc
- no broader helper extraction into `packages/core`
- no extension/setup UX work
- no deterministic GitHub helper-script work
- no second-repo pilot work
- no broad prompt rewrites beyond what the blocker cleanup requires

## Acceptance criteria

- this phase doc contains the imported-asset classification and final phase boundary
- the scoped imported blockers are removed from the edited assets
- `skills/copilot-dev-loop/SKILL.md` preserves its GitHub/Copilot workflow intent while dropping source-repo-specific assumptions
- `agents/review.agent.md` has no hardcoded reviewer identity and no stale plan-path assumption
- `agents/coordinator.agent.md` no longer contains stale `docs/plans/` assumptions and is not over-claimed as generic
- at least one focused regression test guards against reintroducing the known blocker phrases
- the work stays phase-bounded and does not add unnecessary doc or tmp complexity

## Validation approach

- write the blocker-regression test first
- run `node --test test/imported-assets-normalization.test.mjs` during iteration
- add and run `npm run test:assets`
- run `git diff --check`
- do a targeted read-through of the edited markdown assets to confirm workflow behavior was preserved
- record any validation limitations honestly rather than inventing broader checks

## Durable decisions

- for this phase, imported-asset classification lives in this phase doc rather than a second durable inventory document
- temporary planning artifacts should stay minimal and phase-bounded
- the dev-mode follow-up for this phase simplified the default dev-loop artifact surface so optional planning artifacts are no longer generated by default
- dev mode in this repo now requires a dedicated dev retrospective plus at least one bounded skill and/or agent-prompt update before the phase can leave dev mode
- `developer`, `docs`, and `quality` are the first expected candidates for “ready to globalize now,” pending the blocker audit
- `coordinator` should be treated conservatively and is unlikely to be fully generic after this phase
